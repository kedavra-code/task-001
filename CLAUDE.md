# CLAUDE.md

Project instructions and working notes for Claude Code sessions in this repository. Read this first when starting work here.

## Overview

task-001 (formerly "Task Dispatcher") is a React web app for fast operational task triage: capture incoming work, decide what is important, urgent, or still unclear, and keep it visible until it is started or done. It is for individuals and small operational teams that need lightweight action steering rather than a broad project or collaboration suite. Collaboration happens in other tools; this app stays deliberately narrow: categorize, prioritize, and review tasks.

2026-07-24: Workability, Assignee, and Action (and everything that depended on them — the Master Dispatcher setting, dispatch state, Follow-up dates, and the Kanban Delegate column) were removed entirely by user request, along with the edit-card layout restructuring that followed. See the Current Behavior section below for the resulting shape; do not reintroduce these concepts without being asked. The `Open` status label (Kanban column, Status dropdown/badges everywhere via `DISPLAY_VALUE_LABELS["Offen"]`) was also renamed to `Backlog`, and `Started` to `Doing` (`DISPLAY_VALUE_LABELS["Gestartet"]`); the internal values `Offen`/`Gestartet` are unchanged. A `Done` tab was added next to `All` in row 1 of the tab layout (`STATIC_TAB_IDS`), visible in both List and Kanban view; selecting it forces list view off Kanban since Kanban has no done column.

Stack: React 19, TypeScript 6, Vite 8, Supabase (auth + storage), Google OAuth, Vercel hosting, GitHub source control.

## Working Rules

- GitHub `main` is the source of truth. Run `git status --short` and `git pull --rebase --autostash` before doing anything else in every session, unconditionally — not just when a change looks relevant. This project is worked on from more than one computer, so the local checkout may be behind regardless of how the task looks; do not rely on memory of an older local checkout.
- Before every push: run `npm test`, `npm run lint`, and `npm run build`.
- Do not start or check the local dev server (`npm run dev`) as part of normal verification — only if the user explicitly asks for it.
- Before pushing, update `README.md` when user-facing behavior changed, the in-app `...` -> `Docs` panel, and this file (`CLAUDE.md`) when working rules or current behavior changed. Keep them in sync with the app in the same commit — this file drifting from actual behavior makes it actively misleading.
- Commit with a clear message and push directly unless the user says not to.
- After implementing a change, tell the user exactly what to test.
- If a change needs manual action outside the repo — Supabase SQL, Vercel/GitHub settings, OAuth config — call it out explicitly as a separate note with the exact SQL or steps required.
- Apply changes consistently regardless of which machine/session is doing the work: preserve existing safety confirmations, popup-close behavior, documentation updates, and verification/push flow unless the user explicitly changes the rule.

- 2026-07-24: Fixed a data-loss bug where a user's first successful login (after Supabase env vars had been broken/missing, so the app had been running in local-only mode) could wipe their local tasks and settings and cascade-delete the matching Supabase rows. Root cause: the login-time task merge (`mergeLocalExpandedTaskFields`, now renamed `mergeRemoteAndLocalTasks` in `src/App.jsx`) only ever mapped over `remoteTasks` — any task that existed only in local state (never synced) was silently dropped from the merged result. Fix: `mergeRemoteAndLocalTasks` now unions local-only tasks into the merged result instead of dropping them (still enriches matched remote tasks with richer local subtask/comment/tag/predecessor data as before), so first-sync local-only tasks get uploaded via the normal save effect instead of deleted. Known remaining limitation, not fixed here (pre-existing, unrelated to this bug): none of this app's localStorage keys are scoped per user, so switching between different allowed-user Google accounts on the same physical browser can still mix up local state across accounts; this only matters for a shared-browser multi-user scenario, which isn't how the app is normally used.

- 2026-07-24: Fixed a real data-loss bug where done and deleted tasks were being permanently removed from Supabase (confirmed via direct SQL: `select ... where google_status = 'Erledigt' or deleted_at is not null` returned zero rows, even though the tasks existed and reload didn't bring them back). Root cause: `saveRemoteTasks` in `src/App.jsx` computed which Supabase rows to hard-delete by diffing *every* existing Supabase row for the user against whatever tasks happened to be in this browser tab's in-memory `tasks` state at that moment, then deleted anything Supabase had that this tab's memory didn't. That in-memory state is not guaranteed to be a complete, current view of a user's tasks — a tab left open across a Supabase auth token refresh (which changes the `session` object and re-triggers the always-on save effect even though `event === "TOKEN_REFRESHED"` intentionally skips resetting `isLoaded`/re-fetching), a second tab/device that made changes this tab doesn't know about, or any other timing gap would all cause this same full-table diff to wrongly treat real rows as "no longer wanted" and delete them — this is a different, broader instance of the same underlying anti-pattern as the two data-loss bugs above, just triggered by any stale-snapshot save instead of only the first-login merge. Fix: `saveRemoteTasks` no longer diffs against the full remote table at all; it now only hard-deletes rows whose `id` was actually pruned by `pruneExpiredDeletedTasks` (i.e. `isDeletedExpired` was true for that task in the array passed in) — the one case the app is actually supposed to hard-delete for (30-day deleted-task retention). Every other row is left alone and simply gets upserted or ignored; the `existingRows`/`selectError` full-table select this used to require before deleting is gone too. `saveRemoteSubtaskRows`'s own per-task `task_subtasks` delete (scoped to `.in("task_id", taskIds)` of the tasks actually being saved) was already safe and is unchanged — it can never touch subtasks of a task this tab doesn't currently know about.

Leaving or closing the task with unsaved comment edits uses the normal `Save changes?` dialog. New comment textareas grow from one row up to four rows, while existing comment textareas grow with their content up to six rows; both use internal scrolling after that, with save/delete icons using the same right-side layout and matching size as subtask editor icons. Subtasks and comments both render as visually separated read entries in expanded task details. Description filtering/sorting includes comment text. Supabase requires `alter table public.tasks add column if not exists comments jsonb not null default '[]'::jsonb;`; if comments exist and the column is missing, remote save is blocked to avoid silent loss.
