## Current Product Behavior

- The app is a Vite/React task dispatcher with Supabase auth/storage and Vercel deployment.
- Product positioning: Task Dispatcher focuses on fast operational task triage: capture incoming work, decide what is important, urgent, still unclear, self-owned, or delegable, and keep it visible until it is started or done. It is for individuals and small operational teams that need lightweight action steering rather than a broad project or collaboration suite. Collaboration happens in other tools; this app stays deliberately narrow: categorize, prioritize, dispatch, and follow up.
- Google access is managed through Supabase `allowed_users`. `miro@pixelina.me` is the admin and can add/remove allowed Google users inside the app under `...` -> `Users`. Each allowed user has a separate task list and separate synced settings via `user_id`; this is not shared-workspace collaboration.
- Mobile capture is intentionally lightweight: only the fast fields are shown, plus `Assignee` so dispatch ownership can be set immediately on a phone. The mobile assignee combo uses a full-width row and at least a 44px touch target; tapping the input or arrow opens its options.
- Tasks captured without further details remain intentionally unresolved: `When` appears as `clarify`, priority as `prioritize`, assignee as `to decide`, and empty dates/dependencies stay empty.
- `clarify` tasks sort to the top, so lightweight mobile captures are easy to triage later.
- Desktop capture and editing show the fuller set of fields because there is more space and usually more time. `When` and `Prio` are derived from criteria, shown while capturing/editing on desktop and mobile, recalculated from criteria whenever a task is normalized/saved, and are not manually set. `When` answers `When should we act?`; `Prio`/priority answers `How important?`; `Urgency` answers `How long can we still wait?`. `Prio` describes importance and changes slowly; `Urgency` describes time pressure and can change quickly. Examples to preserve: high-priority/low-pressure long-term migration, low-priority/high-pressure certificate expiring in 30 minutes. Priority is Impact × Damage; Urgency is remaining action margin × escalation speed. `WIP` and `Private` are no longer used as `When` filter values; started work uses status `Started`, and private work uses tags such as `Private`.
- Browser capture keeps the full desktop form, but the capture card is content-based and Task, description, tag, and the success message are width-limited so they do not stretch across the whole screen. Phone capture remains responsive/full-width.
- Text fields use lightweight input guidance and field-specific limits: task names wrap in capture/edit, use a wider controlled field while capturing and editing, are not manually resizable, warn at 80 characters, warn stronger at 125, and stop at 250; other multiline text fields scroll internally when needed, but no textarea shows a manual resize handle; subtasks warn at 250 and stop at 1000; comments warn at 500 and stop at the high technical limit of 5000; descriptions have no early warning and stop at the high technical limit of 20000; tags stop at 24; assignee names and Master Dispatcher stop at 60.
- The overview has three configurable navigation rows. Default is row 1 `Upcoming`, row 2 `All` followed by active tag tabs, row 3 `Newest`, `Open`, and `Started`. Any visible tab can be moved across these three rows by drag-and-drop; drag-and-drop surfaces use an open-hand cursor on hover and a closed-hand cursor while dragging; the `...` menu button `Reset tabs` restores the default layout, and `Reset Upcoming order` clears the manual Upcoming drag-and-drop order so the standard sort applies again. The custom layout is stored locally and in `user_settings.tab_layout` when available. The optional due row `Start reached` and `Overdue` is hidden by default and can be toggled in the `...` options menu; hiding it clears an active due-tab filter. Normal task views use the standard task order: `When` (`clarify`, `now`, `later`, `someday`), then `Prio` (`prioritize`, `P1`, `P2`, `P3`), then due date, then task title, unless a column sort is active. `All` plus all non-tag status/due tabs have explanatory tooltips; `Open`, `Newest`, and `Overdue` explain which tasks they include. The `Overdue` tooltip says open or started tasks/subtasks whose `Due` date has been reached, while tag tabs do not get extra explanatory tooltips. `Upcoming` shows reminder tasks globally in persistent order, clears the active tag scope to `All`, and its tooltip describes it as everything that needs immediate attention. Its counter turns red with a white number when tasks are waiting; at `0` it uses normal counter colors. Tasks in `Upcoming` can be reordered by drag-and-drop only while the standard List view is active; dragging near the visible top or bottom edge starts auto-scrolling early, and the order is stored in `user_settings.due_reminder_order`. `Upcoming` cannot be deactivated by clicking it again; clicking `All` or a tag while `Upcoming` is active switches directly to the neutral view with all active tasks in that scope. Other active status tabs can still be clicked again to show all active tasks in the current scope except done/deleted tasks; clicking an active due tab again clears that due filter. `Newest` shows active, not-done tasks in the active scope with the newest effective creation timestamp first, clears due filters on switch, and sorts real new-task `created_at` values before repaired legacy fallbacks. Old tasks with missing, former task-code-derived, or identical Supabase backfill `created_at` values use their start date once as creation timestamp. Tasks with neither value appear after dated tasks. `Done` and `Deleted` are opened from the `...` menu and apply to the active scope, so done/deleted tasks can be viewed globally from `All` or per tag from e.g. `#Private`. Capture is opened through the header plus icon, not a tab; fresh browser and phone starts open their configured start tab (`All` by default, or `Upcoming` if selected); capture is opened through the plus icon. Clicking/touching the `Task Dispatcher` title always jumps to neutral `All` without additional status or due filters. Switching status/menu views resets filters and closes active editing. The filter popup still includes an editable status dropdown (`All`, `Open`, `Started`) for ad hoc filtering and sorting inside the current scope; `Done` and `Deleted` stay in the scoped `...` menu views. Changing status or due filters activates the matching visible tabs where such tabs exist.
- Top-level and status tab controls are individually framed buttons, not only a single framed tab container.
- Tags are managed separately in the `...` options menu under `Tags`, stored in Supabase user settings, limited to 10 catalog tags, and remain available until deleted. The dialog shows the current catalog count, e.g. `3 of 10`. Existing tags can be renamed there with the pencil icon and saved with the disk icon; renaming updates task assignments, selected tag tabs, drafts, and the active scope. Tasks can have one tag from that catalog. Tag assignment is available while capturing and editing on mobile and desktop through a compact selection popup; selecting another tag replaces the previous one, `No tag` removes it, and empty tag fields show `-` and open the same popup. Assigning a tag to a task automatically activates the matching tag tab. Deleting a catalog tag removes it from task assignments and selected tag tabs. Existing tags can also be selected/deselected as `Tag-Tabs` in the `...` -> `Tags` dialog. This selection and order apply on mobile and desktop; tag tabs can be reordered by drag-and-drop in the top tag row or Tags dialog. Tag scopes include their own `Open`, `Started`, and `Newest` views, plus scoped `Done` and `Deleted` via the `...` menu; `Upcoming` is a global reminder view and resets the tag scope to `All`.
- Top-level tabs show compact counters for `All`, `Upcoming`, and each selected tag tab. `All` and tag tabs count only active tasks in `Open` and `Started`; `Upcoming` counts global pending reminder tasks and turns red/white when greater than zero. The `Newest` status tab counts the same active, not-done scope. Done and deleted tasks are counted only in the scoped `Done` and `Deleted` entries in the `...` menu.
- Status and due tabs show compact counters for the active top-level scope; the scoped done and deleted counters are shown only in the `...` menu entries. The title area does not repeat open/started/done/deleted/due summaries; due and overdue counts belong to tabs and `Upcoming`.

Single-option rows keep the option in the left table cell instead of spanning the full menu. In the browser, the menu opens from the left edge of the `...` button so it stays inside the viewport. Menu controls have concise tooltips explaining each option.

Deleting keeps the previous view instead of jumping to `Deleted`, and resets that view's filters so remaining tasks stay visible. Expired deleted tasks are physically removed on the next save/load cycle. Supabase requires `alter table public.tasks add column if not exists deleted_at date;`.

The message is cleared as soon as the user opens or starts capturing another task, unless they click the task-code link itself. Only the task name is required and explicitly marked as `required`; criteria dropdowns remain optional and must visibly start unset for each new capture because the app can derive `When` and `Prio` from missing or placeholder values. Other task links jump to the scope and view where the task is visible; for tagged tasks with an active tag tab, this is the matching tag scope plus the task's current view, and done/deleted tasks jump to the scoped `Done` or `Deleted` menu view. After editing, the app restores the previously active tab, tag scope, and filters. Pure save while staying in edit mode keeps the edited task visible even if a non-completion status change would otherwise remove it from the active filter. Completing from edit mode closes the task and returns to the previous view; deleting keeps the previous view, while restoring and direct task links may still navigate to the matching visible view.
- The browser and phone task overview has a persistent free-text search above the task list, and the same field remains visible while capturing a new task. During active search, scope/status/due tabs stay visible but no tab is highlighted as active. Multiple words may match across fields.

## Data Model Notes

- Supabase stores multiple predecessors in `depends_on_task_ids`.
- Supabase stores subtasks in `task_subtasks` and still writes the legacy `tasks.subtasks` text array during the transition. `task_subtasks` is preferred on load when rows exist; duplicate rows for the same `(user_id, task_id, position)` are collapsed in the app. Identical subtask text is allowed because two checklist items may intentionally have the same wording. The unique index prevents new position duplicates once run in Supabase. `tasks.subtasks` remains the fallback for older data/backups.
- Supabase stores task tags in the `tags` text array.
- Supabase stores task comments in `tasks.comments` as JSONB. Required schema: `alter table public.tasks add column if not exists comments jsonb not null default '[]'::jsonb;`.
- Supabase stores task creation timestamps in `tasks.created_at`; new tasks write and keep the current timestamp and sort before repaired legacy fallbacks. Old tasks without one, with the former task-code-derived legacy timestamp, or with an identical Supabase backfill timestamp shared by several old tasks use their start date once as creation timestamp on load/save. Tasks without both creation timestamp and start date get an old technical fallback so they sort after dated tasks. The `Newest` tab sorts by newest effective creation timestamp first, with start date as fallback and tie-breaker. Required schema: `alter table public.tasks add column if not exists created_at timestamptz not null default now();` plus `create index if not exists tasks_created_at_idx on public.tasks (created_at);`.
- Supabase stores synced per-user UI settings in `user_settings`; currently this is used for the managed tag catalog in `available_tags`, selected tag tabs in `selected_tag_tabs`, custom tab layout in `tab_layout`, browser compact overview mode in `browser_compact_view`, badge column counts for list/edit/browser-Kanban cards in `card_badge_columns`, separate browser/phone default list/Kanban view modes in `default_view_mode` / `default_view_mode_mobile`, separate browser/phone default start tabs in `default_start_tab` / `default_start_tab_mobile`, Kanban column visibility in `kanban_columns`, Master Dispatcher in `master_dispatcher_name`, tooltip preference in `tooltips_enabled`, and separate browser/phone dark-mode preferences in `dark_mode_browser` / `dark_mode_mobile`. Active tab, tag scope, search text, filters, column sorts, and header icon List/Kanban switches are session-only working state and must not be persisted/restored across app restarts; persistent default List/Kanban view must only change from the options menu.
- Supabase stores allowed Google users in `allowed_users`; RLS lets each user read their own access row and lets the admin manage all access rows.
- Supabase stores the automatic done date in `completed_at`.
- Supabase stores the soft-delete date in `deleted_at`.
- If required columns are missing, the app blocks remote saving instead of silently losing multi-predecessor, comment, or subtask data.
- The app also keeps a local browser backup to protect against accidental data loss.
- Supabase schema file includes auth hook functions, `task_subtasks`, and extra synced settings columns. The user ran the schema and enabled/tested the Auth Hooks on 2026-06-04.
- Future schema changes should be executed directly by Codex when access is available; otherwise provide a clear manual-action note with the exact SQL and tell the user it must be run before the deployed feature can persist correctly.
